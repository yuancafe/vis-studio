# Brand Identity Process Synthesis

This document summarizes the end-to-end workflow embodied by the V3 skill suite.

## Core thesis

Brand identity work is not “make a logo and choose colors.” It is a staged process that moves from diagnosis to strategy, from strategy to visual translation, from visual translation to an engineered identity system, from identity system to application production, and from there into governance and reuse.

## Public workflow

### Phase 1. Discovery and strategy

First determine:

- what exists already
- what is broken
- what must be preserved
- what business change the identity must support
- what level of deliverable is actually needed
- how much application work is likely in scope

Never begin with aesthetic preference alone.

### Phase 2. Concept exploration

Translate strategy into a curated range of visual territories.

Define:

- perception goals
- category codes to keep or reject
- emotional temperature
- formal tension
- candidate style playbooks

Present 2-4 territories, not a dump of possibilities.

### Phase 3. Identity system production

Engineer the chosen concept into a robust core system.

Define:

- responsive logo logic
- monochrome and reverse behavior
- typography hierarchy
- color architecture
- imagery direction
- supporting graphic system
- foundational application proofs

This phase stops at core system quality and proof, not at mass collateral output.

### Phase 4. Brand application system

Turn the core system into concrete touchpoint applications.

Handle:

- application scope planning
- scope bands
- production-route classification
- template-ready items
- mockup-led items
- manual-vector briefs

This is the explicit “visual ecosystem development” layer that V2 under-modeled.

### Phase 5. Governance and rollout

Finalize the usable operating system.

Package:

- Brand Identity Guideline
- approved rules and misuse examples
- final brand pack
- asset manifest
- access model
- governance ownership
- rollout priorities

## Brand-pack lifecycle

The machine-readable `brand-pack` is a first-class artifact across phases.

### Foundation after identity production

Create the initial pack with:

- `brand-guideline`
- `brand-tokens`
- `logo-system`
- `style-playbook-selection`

### Enrichment during application production

Add:

- `application-recipes`
- scope matrix
- bundle presets
- production-route data

### Finalization during governance

Add:

- `asset-manifest`
- publication metadata
- wrapper-skill handoff
- governance metadata

## Standards

### Strategy standards

- Define the audience specifically
- Define claims with proof
- Define personality with boundaries
- Define differentiation against real alternatives
- Secure leadership alignment on the strategic brief before entering heavy visual work

### Concept standards

- Use moodboards or stylescapes to align on direction before overbuilding final executions
- Prefer milestone reviews over continuous unstructured commentary
- Present a curated set of options, not a large undecided pile
- Judge logo form in black and white before color is introduced
- Narrow concept work through explicit fit and avoid signals

### Identity-system standards

- Treat logo as one component, not the entire brand
- Build color roles, not just color moods
- Build typography hierarchy, not only pairings
- Define imagery with enough specificity that another team can brief or source it
- Build reusable graphic rules, not isolated flourishes
- Limit typography to a controlled set, typically 2-3 families
- Specify colors across relevant formats, including digital and print systems when needed
- Check accessibility and small-scale use as part of approval, not as an afterthought

### Application standards

- Classify every application item by production route
- Separate scalable template work from vector-heavy custom work
- Do not let a large scope list flatten into one undifferentiated batch
- Use mockups to explain context and fit, not as a substitute for rules
- Treat scope bands explicitly:
  - `Core` = 1-10 items
  - `Growth` = 11-30 items
  - `System` = 31-100 items
  - `Enterprise` = 100+ items, batched and prioritized

### Documentation standards

- Explain why the rule exists
- Show correct and incorrect use
- Document only approved decisions
- Match deliverables to actual project scope
- Treat guidelines as a living operating document, not only a static presentation artifact

### Governance standards

- Keep assets in a shared source of truth or DAM
- Assign ownership for reviewing brand compliance
- Train internal teams and external vendors before full rollout
- Publish the wrapper-skill handoff only after pack data is stable

## Taboos

- Beginning with “what styles do you like?” and stopping there
- Preserving weak assets out of nostalgia
- Rebuilding everything when refinement would preserve useful equity
- Using archetypes, values, or moodboards as substitutes for real diagnosis
- Creating a “full system” that has never been tested on real touchpoints
- Confusing a polished presentation with a usable operating system
- Treating application work as a vague appendix instead of a scoped phase
- Pretending Illustrator-heavy vector work is fully automated when it is not
- Launching a new identity externally before internal alignment is ready

## Scope boundaries

The suite supports phased brand identity work. It does not assume every project needs the same depth.

Use phased outputs:

- strategy only
- concept directions
- core identity
- identity plus application system
- guideline and rollout pack
- wrapper-skill handoff

Treat deferred work explicitly as deferred work.
